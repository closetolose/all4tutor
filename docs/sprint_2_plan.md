## Стандарты и ограничения

- **Стандарты:** следовать правилам из [`.cursor/rules/ui-standards.mdc`](../.cursor/rules/ui-standards.mdc) (Django, Vanilla JS, UI/UX). В эпике они указаны как «frontend-django-standards» — в проекте им соответствует файл `ui-standards.mdc`.

## Блок A: UI Fixes & Dark Mode

### Задача 1 — Тёмная тема (Desktop): аудит и замена хардкод-цветов на var(...)

**Цель:** Все ключевые цвета на десктопе заданы через CSS-переменные; тёмная тема корректна и без «протекающих» HEX.

**Затронутые файлы/экраны:**
- `static/css/style.css` — переменные в `:root` и `[data-theme="dark"]`
- Шаблоны в `core/templates/core/`, при необходимости `core/templates/mobile/`, где остались жёстко заданные цвета

**Критерии приёмки:**
- В CSS и шаблонах нет хардкод HEX/RGB для фона, текста, границ (кроме объявления переменных в `:root` / `[data-theme="dark"]`).
- Тёмная тема переключается и сохраняется (например через `data-theme` и localStorage).
- Контраст в тёмной теме достаточный (де-факто WCAG AA где применимо).

**Подзадачи:**
1. Аудит: поиск всех вхождений HEX/RGB в `style.css` и в шаблонах (inline-style, классы).
2. Введение/дополнение переменных и блока `[data-theme="dark"]`.
3. Замена найденных цветов на `var(--...)`.

**Цель:** Надпись «Сортировка» не переносится и не ломает верстку (nowrap или flex/ширина).

**Затронутые файлы/экраны:**

- `static/css/style.css` — классы, связанные с блоком сортировки

**Критерии приёмки:**

- Блок сортировки не выходит за границы контейнера (нет горизонтального overflow).
- Решение: `white-space: nowrap` и/или flex с подходящей шириной/мин-шириной.

**Затронутые файлы/экраны:**

- Проверка `{% if %}` и контекста из `core/views.py` (view карточки ученика)

**Критерии приёмки:**
- Кнопки «Редактировать ДЗ» и «Удалить ДЗ» (или аналоги) видны там, где по ТЗ они должны быть (роль: репетитор/владелец).
- Условия в шаблоне (`{% if %}`) соответствуют роли и владельцу связи ученик–репетитор.

**Подзадачи:**

## Блок C: Features

### Задача 9 — Модалка посещаемости: вынести отметку из таблицы; Desktop — модалка, Mobile — Bottom Sheet

**Цель:** Отметка о посещаемости вынесена в отдельное окно; на десктопе — модалка по центру, на мобильных — Bottom Sheet (шторка снизу) с анимацией и закрытием по свайпу/кнопке.

**Затронутые файлы/экраны:**
- Шаблоны, где отображается таблица занятий/посещаемость (например уроки, журнал): `core/templates/core/`, `core/templates/mobile/`
- Новый фрагмент/модальное окно для формы посещаемости
- `static/css/style.css` — стили модалки и Bottom Sheet, медиа-запросы

**Критерии приёмки:**
- В таблице занятий больше нет инлайн-редактирования посещаемости; вместо него — действие (кнопка/иконка), открывающее окно.
- **Desktop (≥ 768px):** окно — модалка по центру экрана (overlay + центрированный блок).
- **Mobile (< 768px):** окно реализовано как Bottom Sheet: выезжает снизу (`transform: translateY`), закрытие по кнопке и при желании по свайпу вниз.
- Анимация открытия/закрытия через CSS (transition по `transform` или opacity).

**Подзадачи:**
1. Вынести форму/поля отметки посещаемости в отдельный HTML-фрагмент (модалка/Bottom Sheet).
2. Подключить открытие/закрытие окна по клику (Vanilla JS), передача id занятия/урока.
3. **Разработка адаптивного CSS (Media Queries) для экрана < 768px:**
   - Для ≥ 768px: модалка по центру (fixed overlay, flex center).
   - Для < 768px: контейнер шторки — fixed bottom 0, left 0, right 0; контент с `transform: translateY(0)` при открытии и `translateY(100%)` при закрытии; transition для плавности.
4. Реализация Bottom Sheet: закрытие по кнопке «Закрыть»/крестик; опционально — обработка свайпа вниз (touch events) для закрытия.
5. Сохранение данных посещаемости через существующий backend (без изменения логики вне задачи при необходимости).

### Задача 11 — Аналитика: фильтры по ученику и репетитору; раздел для учеников и безопасность

**Цель:** Фильтры по ученику и репетитору; отдельный раздел аналитики для учеников, где они видят только свои результаты; гарантированная фильтрация по `request.user` на бэкенде.

**Затронутые файлы/экраны:**
- `core/views.py` — представления аналитики: фильтрация QuerySet по ученику/репетитору; для роли «ученик» — только данные текущего пользователя
- Шаблоны аналитики/результатов: `core/templates/core/results.html`, `core/templates/mobile/results.html` (и связанные)
- URL-маршруты и формы фильтров

**Критерии приёмки:**
- Репетитор может фильтровать аналитику по ученику и по репетитору (если применимо).
- Ученик видит раздел аналитики/результатов только со своими данными.
- **Безопасность:** во всех представлениях аналитики QuerySet фильтруется по `request.user`: для ученика — только записи, где ученик = `request.user`; запрет доступа к чужим данным (возврат 403 или пустой результат при попытке доступа по id другого пользователя).

**Подзадачи:**
1. Добавить фильтры (форма/GET-параметры) по ученику и репетитору в представлениях для репетитора/админа.
2. Открыть раздел аналитики для учеников: отдельный view или ветвление по роли; в шаблоне показывать только разрешённые блоки.
3. **Шаг по безопасности:** во всех view аналитики/результатов:
   - Для учеников: `QuerySet.filter(..., user=request.user)` (или аналог по модели); не использовать переданный id ученика для доступа к чужим данным.
   - Проверка прав: если в URL или в параметрах приходит id другого пользователя — не возвращать его данные ученику; при необходимости возвращать 403.
4. Обновить шаблоны: отображение фильтров для репетитора, упрощённый вид для ученика.

## Блок D: Admin

### Задача 12 — Скрыть модели User и Group в стандартной админке Django

**Цель:** В стандартной админке (`/admin/`) модели User и Group отключены от отображения через `admin.site.unregister()`; акцент остаётся на кастомной панели (`/dashboard/admin/`).

**Затронутые файлы/экраны:**
- `core/admin.py` — место регистрации/импорта моделей Django; вызов `admin.site.unregister(User)` и `admin.site.unregister(Group)` после импорта из `django.contrib.auth`.

**Критерии приёмки:**
- При входе в `/admin/` в списке приложений/моделей нет User и Group (или они не кликабельны и не ведут на стандартные формы).
- Кастомная админ-панель `/dashboard/admin/` продолжает работать как раньше.
- Никакой другой функционал стандартной админки не сломан (остальные зарегистрированные модели при необходимости остаются).

**Подзадачи:**
1. В `core/admin.py` (или в том файле, где конфигурируется `admin.site`): импортировать `User` и `Group` из `django.contrib.auth.models`.
2. Вызвать `admin.site.unregister(User)` и `admin.site.unregister(Group)` после того, как они зарегистрированы по умолчанию (обычно после импорта `django.contrib.admin` и регистраций приложения).
3. Проверить вход в `/admin/` и в `/dashboard/admin/`.

## Верификация (Verifier)

Рекомендуемый порядок передачи на проверку: **по блокам** — после завершения блока A → Verifier; затем B → Verifier; затем C → Verifier; затем D → Verifier.

Рекомендуемые критерии для Verifier:

| Критерий | Описание |
|----------|----------|
| **Mobile Touch Test** | Все интерактивные элементы (кнопки, ссылки, поля) на ширине ≤ 768px имеют размер не менее 44px (лучше 48px по высоте для полей); нет случайных срабатываний. |
| **Overflow Test** | На мобильных нет горизонтального overflow у body/основного контейнера; блоки с длинным текстом (сортировка, палитра, таблицы) не вылезают за экран. |
| **Security Test** | Для задачи 11: от имени ученика проверить, что в аналитике/результатах отображаются только его данные; попытки доступа к данным другого пользователя (например подмена id в URL) возвращают 403 или пустой результат. |
| **Admin Test** | В `/admin/` модели User и Group не отображаются; кастомная панель `/dashboard/admin/` открывается и функционирует. |

Дополнительно: проверка тёмной темы (задача 1), видимость кнопок на карточке ученика (задача 3), работа модалки и Bottom Sheet на разных ширинах (задача 9).
