# План разработки кастомной панели администратора (Superuser Dashboard)

**Важно:** Реализуется только кастомная панель внутри основного сайта (свои Views, Templates, Vanilla JS). Стандартная Django Admin (`/admin/` или `secretplace/`) не кастомизируется.

---

## Контекст проекта

- **Модели:**  
  - **Users** — профиль пользователя (связь с `AUTH_USER_MODEL` через `OneToOneField`). Поля: `id`, `role` (tutor/student), `first_name`, `last_name`, `patronymic`, и др. Статус активности — у Django User: `user.is_active`.  
  - **ConnectionRequest** — связь репетитор–ученик: `tutor`, `student`, `status` (pending/confirmed/rejected/archived).  
  - **UnlinkRequest** — заявка на открепление: `student`, `tutor`, `status` (pending/approved/rejected), `created_at`, `reviewed_at`, `reviewed_by`. Поля «причина» в модели нет (при необходимости — отдельная доработка).  
- **При одобрении заявки:** переводим соответствующую связь `ConnectionRequest` в `status='archived'`. Модели `Lesson`, `LessonAttendance`, `Transaction`, `PaymentReceipt` не удаляются.  
- **Деструктивные действия:** выполнять в `transaction.atomic()`.

---

## Шаг 1: Роутинг, защита, базовые Views и ссылка на панель

### 1.1 URL

- Префикс: `/dashboard/admin/`.  
- Примеры:  
  - `/dashboard/admin/` — главная панели (редирект на список пользователей или общий дашборд).  
  - `/dashboard/admin/users/` — таблица пользователей.  
  - `/dashboard/admin/unlink-requests/` — таблица заявок на открепление.

Подключить в `tutor_project/urls.py` через `path('dashboard/admin/', include(...))` или явно перечислить пути в `urls.py` (например, `path('dashboard/admin/', views.dashboard_admin_index)` и т.д.).

### 1.2 Защита доступа

- Все View панели администратора защищать проверкой суперпользователя:  
  - FBV: `@user_passes_test(lambda u: u.is_superuser)`.  
  - CBV: `UserPassesTestMixin` с `test_func(self): return self.request.user.is_superuser`, плюс `LoginRequiredMixin`.  
- При отсутствии прав — редирект на главную или 403 (на выбор, единообразно).

### 1.3 View-функции/классы (минимальный набор для шага 1)

- **Главная панели:** например `dashboard_admin_index` — редирект на `dashboard/admin/users/` или рендер страницы с сайдбаром и контентом «Добро пожаловать» с ссылками на разделы.  
- **Список пользователей:** `DashboardUserListView` (или FBV `dashboard_admin_users`) — передаёт в шаблон список/QuerySet пользователей (через профили `Users` и связь с `User` для `is_active`).  
- **Список заявок на открепление:** `DashboardUnlinkRequestListView` (или FBV `dashboard_admin_unlink_requests`) — передаёт `UnlinkRequest.objects.all()` (или с фильтром/пагинацией).

Импорты: из `core.models` — `Users`, `UnlinkRequest`, `ConnectionRequest`; для проверки блокировки — `User` (AUTH_USER_MODEL).

### 1.4 Ссылка на панель в шаблонах

- В сайдбаре основного сайта (например, `core/templates/core/base.html`): показывать пункт меню «Панель администратора» (или «Админ-панель») с ссылкой на `/dashboard/admin/` **только если** `request.user.is_superuser`.  
- Аналогично при наличии отдельного сайдбара для mobile — показывать ссылку только суперпользователю.  
- Не добавлять ссылок на стандартную админку Django в рамках этого эпика; при необходимости ссылка на `secretplace/` может оставаться только у тех, кто знает URL.

## Шаг 2: Верстка страниц панели — таблицы, сайдбар, модалки

### 2.1 Layout и сайдбар админки

- Отдельный базовый шаблон для админки, например `core/templates/core/dashboard_admin/base.html`, расширяющий общий `core/base.html` или подключающий общие стили (тёмная тема, CSS-переменные, адаптивность).  
- В контенте панели — свой сайдбар с разделами:  
  - «Пользователи» → `/dashboard/admin/users/`.  
  - «Заявки на открепление» → `/dashboard/admin/unlink-requests/`.  
- Контент справа: блок для таблиц и действий.

### 2.2 Таблица пользователей

- Колонки: ID (профиля или User), Роль, Имя (ФИО из профиля), Email (из `User.email`), Дата регистрации (например `User.date_joined`), Статус (активен/заблокирован — из `User.is_active`).  
- Поиск: по email, имени, ID (GET-параметры).  
- Фильтр по роли: все / репетитор / ученик (GET).  
- Контейнер таблицы с `overflow-x: auto` для горизонтальной прокрутки на малых экранах (в т.ч. 375px).  
- В каждой строке — блок «Действия»:  
  - Заблокировать / Разблокировать (переключение `User.is_active`).  
  - Удалить — только через модальное окно (не `confirm()`): в модалке обязательный ввод слова «УДАЛИТЬ» и кнопка подтверждения.  
- Пагинация: Django `Paginator` (например по 25–50 записей на страницу).

### 2.3 Таблица заявок на открепление

- Колонки: кто подал (ученик), от кого (репетитор), причина (если в модели нет — вывести «—» или не показывать колонку до появления поля), дата заявки, статус.  
- Контейнер с `overflow-x: auto`.  
- Пагинация через `Paginator`.  
- В строке для заявок со статусом «Ожидает рассмотрения» — кнопки «Одобрить» и «Отклонить» (обработка через Ajax на шаге 3).

### 2.4 Модальное окно удаления пользователя

- Стилизованное модальное окно (в духе текущего дизайна): заголовок, текст предупреждения, поле ввода, куда пользователь должен ввести слово «УДАЛИТЬ», кнопки «Отмена» и «Подтвердить».  
- Кнопка «Подтвердить» активна только при точном совпадении введённого текста с «УДАЛИТЬ».  
- Деструктивное действие (удаление User и связанного профиля) выполнять в `transaction.atomic()` на бэкенде.

## Шаг 3: JS-логика — Ajax, JSON, Toast

### 3.1 Общие требования

- Запросы: `fetch` с заголовком `X-CSRFToken` (значение из cookie `csrftoken` или из тега `{% csrf_token %}`).  
- Метод для изменений: `POST`.  
- Ответы бэкенда — JSON (например `{"success": true, "message": "..."}` или `{"success": false, "error": "..."}`).

### 3.2 Блокировка/разблокировка пользователя

- Endpoint, например: `POST /dashboard/admin/users/<id>/toggle-active/` (или по `user_id` Django User).  
- View: проверка `is_superuser`, получение пользователя, переключение `user.is_active`, сохранение, возврат JSON.  
- В JS: по клику на «Заблокировать»/«Разблокировать» — `fetch` → обновление строки таблицы или перезагрузка данных; показ Toast об успехе/ошибке.

### 3.3 Удаление пользователя

- Endpoint: `POST /dashboard/admin/users/<id>/delete/` с телом, содержащим подтверждение (например `{"confirm": "УДАЛИТЬ"}`).  
- View: проверка `is_superuser`, проверка тела запроса, выполнение удаления в `transaction.atomic()` (удаление профиля `Users` и/или `User` в зависимости от связей).  
- Ответ JSON. На фронте после успеха — закрыть модалку, убрать строку из таблицы или перезагрузить список, показать Toast.

### 3.4 Одобрение/отклонение заявки на открепление

- Endpoints:  
  - `POST /dashboard/admin/unlink-requests/<id>/approve/`  
  - `POST /dashboard/admin/unlink-requests/<id>/reject/`  
- View approve: в `transaction.atomic()` — обновить заявку (status='approved', reviewed_at, reviewed_by), найти `ConnectionRequest` по student/tutor со status='confirmed' и перевести в 'archived'. Удалять Lesson, PaymentReceipt, Transaction не нужно.  
- View reject: обновить заявку (status='rejected', reviewed_at, reviewed_by).  
- Ответы — JSON. В JS по клику на «Одобрить»/«Отклонить» — `fetch` → обновление строки или списка, Toast с результатом.

### 3.5 Toast-уведомления

- Реализовать простой Toast (Vanilla JS + CSS): показ сообщения об успехе/ошибке на несколько секунд, в стиле текущей темы (тёмная тема, CSS-переменные). Можно один общий фрагмент/шаблон для админки.

## Чек-лист по ограничениям (для Orchestrator / код-ревью)

- [ ] Не вносить изменения в стандартную Django Admin (сайт админки и его шаблоны).  
- [ ] Все URL/Views панели защищены `is_superuser`.  
- [ ] Ссылка на панель видна только при `request.user.is_superuser`.  
- [ ] Деструктивные операции (удаление пользователя, одобрение заявки с изменением связи) в `transaction.atomic()`.  
- [ ] При одобрении заявки на открепление не удаляются уроки и чеки оплаты.  
- [ ] Удаление пользователя только через модалку с вводом «УДАЛИТЬ», без использования `confirm()`.

---

## Рекомендуемый порядок реализации

1. Шаг 1 — URL, миксин/декоратор, три View (index, users list, unlink list), ссылка в сайдбаре.  
2. Шаг 2 — базовый шаблон админки, сайдбар админки, таблицы (пользователи, заявки), модалка удаления, пагинация и фильтры.  
3. Шаг 3 — API endpoints для toggle-active, delete, approve, reject и JS (fetch, Toast, обновление UI).

После выполнения плана панель будет доступна по адресу `/dashboard/admin/` только суперпользователям и полностью реализована кастомными Views, шаблонами и Vanilla JS в стиле основного сайта.
