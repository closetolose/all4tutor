# Архитектура: цветовая индикация репетитор/группа

Документ описывает архитектурные решения для фичи «кастомный цвет для сущностей Репетитор/Группа» в All4Tutors (Django, HTML/CSS, Vanilla JS).

---

## 1. Модель данных

### 1.1 Кто и что раскрашивает

- **Ученик** видит уроки и ДЗ от нескольких **репетиторов** и по нескольким **группам**. Ему нужен цвет: «репетитор A», «репетитор B», «группа X», «группа Y».
- **Репетитор** видит уроки и ДЗ по своим **ученикам** и **группам**. Для быстрого различения ему достаточно цвета по **группе** (индивидуальные уроки можно не красить или наследовать цвет «без группы»).

Итого:
- Связь **User (ученик) ↔ Репетитор** уже есть: `ConnectionRequest` (tutor, student, status). Цвет «какой цвет ученик назначил этому репетитору» логично хранить там.
- Связь **User ↔ Группа**: ученик — через `StudyGroups.students` (M2M), репетитор — владелец группы. Отдельной таблицы «заявки на группу» нет. Нужна отдельная сущность для хранения цвета группы для пользователя.

### 1.2 Где хранить

| Сущность | Хранение | Пояснение |
|----------|----------|------------|
| Цвет репетитора (для ученика) | Поле в существующей модели | `ConnectionRequest.color_hex` (CharField, max_length=7, null=True, blank=True). Одна запись = одна связь ученик–репетитор; цвет задаёт ученик. |
| Цвет группы | Новая связующая модель | Отдельная модель: пользователь + группа + цвет. Репетитор задаёт цвет своим группам, ученик — группам, в которых состоит. |

### 1.3 Новая модель: цвет группы для пользователя

```python
# core/models.py

class UserGroupColor(models.Model):
    """Цветовая метка группы для пользователя (репетитор или ученик)."""
    user = models.ForeignKey(Users, on_delete=models.CASCADE, related_name='group_colors')
    group = models.ForeignKey(StudyGroups, on_delete=models.CASCADE, related_name='user_colors')
    color_hex = models.CharField(max_length=7, blank=True, null=True)  # например #6366f1

    class Meta:
        db_table = 'user_group_colors'
        unique_together = ('user', 'group')
```

- **Валидация доступа при сохранении:** репетитор — только свои группы (`group.tutor == user`); ученик — только группы, в которых он состоит (`user in group.students`).
- **Формат цвета:** `#RRGGBB` (7 символов). В шаблонах и CSS использовать как есть; при пустом/null — индикация не применяется.

### 1.4 Изменение существующей модели

- В **ConnectionRequest** добавить поле:
  - `color_hex = models.CharField(max_length=7, blank=True, null=True, db_column='color_hex')`
- Цвет задаёт **ученик** (student) для репетитора (tutor). При отображении расписания/ДЗ ученика цвет берётся из `ConnectionRequest`, где `tutor` = репетитор урока/ДЗ и `student` = текущий пользователь.

## 2. Бэкенд: передача цвета в шаблоны без N+1

### 2.1 Откуда брать цвет для урока

- **Урок** имеет `tutor` и опционально `group`.
- **Для ученика:** цвет урока = цвет репетитора (из ConnectionRequest по student=текущий user, tutor=lesson.tutor) или цвет группы (из UserGroupColor по user=текущий user, group=lesson.group). Если урок групповой — приоритет можно отдать цвету группы, иначе — цвету репетитора.
- **Для репетитора:** цвет урока = цвет группы (UserGroupColor по user=текущий user, group=lesson.group); для индивидуального урока — можно не показывать цвет или оставить нейтральный.

### 2.2 Стратегия загрузки (избежание N+1)

1. **Список уроков (index, load_more_lessons):**
   - Для **ученика:** один раз загрузить словарь «tutor_id → color_hex» по подтверждённым ConnectionRequest (student=user) и словарь «group_id → color_hex» по UserGroupColor (user=user, group_id in list lesson.group_id).
   - Для **репетитора:** словарь «group_id → color_hex» по UserGroupColor (user=user, group_id in list lesson.group_id).
   - Передать в контекст/шаблон либо два словаря `tutor_colors` и `group_colors`, либо для каждого урока уже подобранное значение `lesson_display_color` (в view или через аннотацию — см. ниже).

2. **Аннотация/подсчёт в QuerySet:**
   - Уроки уже загружаются с `prefetch_related('subject', 'group', 'tutor', ...)`. Дополнительно:
     - Для ученика: после получения списка `lessons` собрать `tutor_ids` и `group_ids`, одним запросом выбрать все `ConnectionRequest` (student=user, tutor_id__in=tutor_ids, status='confirmed') с полями `tutor_id`, `color_hex` и все `UserGroupColor` (user=user, group_id__in=group_ids). Построить в Python словари и в контекст передать их плюс `lessons`.
   - В шаблоне для каждого урока: `lesson.tutor.id` и `lesson.group.id` уже доступны; цвет взять из переданного словаря по ключу.

3. **Домашние задания (my_assignments):**
   - У ученика ДЗ привязаны к репетитору (`hw.tutor`). Достаточно один раз загрузить словарь tutor_id → color_hex по ConnectionRequest (student=user) и передать в контекст как `tutor_colors`. В шаблоне для каждого `hw` брать `tutor_colors[hw.tutor.id]`.

4. **Итог:** Никаких обращений к БД внутри цикла в шаблоне. Вся логика — в view: один–два дополнительных запроса с фильтром по списку id, построение словарей, передача в контекст.

### 2.3 API сохранения цвета

- **Репетитор (ConnectionRequest):** обновление цвета делается от имени ученика. То есть endpoint должен проверять, что текущий пользователь — именно `connection.student`, и тогда `connection.color_hex = request.POST.get('color_hex') or None` и `connection.save()`.
- **Группа (UserGroupColor):** view проверяет доступ (репетитор — владелец группы, ученик — в группе), затем `get_or_create(user=request.user.profile, group=group)` и обновление `color_hex`.
- Удобно сделать два небольших view (или один с параметром типа сущности): например `update_tutor_color(request, connection_id)` и `update_group_color(request, group_id)`, вызываемые через POST (форма или AJAX).

## 3. Фронтенд: применение цвета

### 3.1 Принцип

- Цвет не переопределяет глобальные переменные темы (`--tg-*`, `--primary`), а задаётся локально на контейнере карточки/строки.
- Рекомендуемый приём: задать на элементе CSS-переменную, например `--custom-entity-color`, а в общих стилях использовать её для рамки и фона с низкой непрозрачностью.

### 3.2 Передача значения в шаблон

- В контексте для каждого урока/ДЗ передавать вычисленный `display_color` (hex или null).
- В шаблоне для строки/карточки:
  - Если `display_color` задан:
    - `style="--custom-entity-color: {{ display_color }};"` на `<tr>` или обёртке карточки.
  - Если не задан — не выводить переменную (или вывести пусто), тогда класс «индикатора» не применяется или использует прозрачный цвет.

### 3.3 CSS

- В `static/css/style.css` (и при необходимости в мобильных стилях в `mobile/base.html` или общем блоке) определить класс, например `.entity-color-indicator`:
  - `border-left: 4px solid var(--custom-entity-color, transparent);`
  - `background: color-mix(in srgb, var(--custom-entity-color, transparent) 12%, transparent);`  
  или для старых браузеров: `background: rgba(var(--custom-entity-rgb, 0,0,0), 0.12);` (если передавать RGB отдельно).
- Карточка/строка должна иметь этот класс и установленную переменную `--custom-entity-color` только когда цвет задан. Текст внутри остаётся на своём фоне — контраст не страдает.
- Для тёмной темы те же 10–15% от hex дают приемлемый вид; при необходимости можно в `[data-theme="dark"]` слегка уменьшить opacity (например 8–10%).

### 3.4 Где именно вешать класс и переменную

- **Десктоп, таблица расписания:** `<tr class="entity-color-indicator" style="...">` в `core/lessons_rows.html`.
- **Мобильная версия, карточка урока:** на `.lesson-card` в `mobile/index.html` добавить класс и `style="--custom-entity-color: ..."`.
- **Список ДЗ (десктоп и мобильный):** на строку таблицы или обёртку карточки ДЗ — тот же класс и переменная от `hw.tutor` (ученик) / при необходимости от группы.

### 3.5 Форма выбора цвета (настройки)

- Использовать нативный `<input type="color">` и/или фиксированную палитру из 6–8 цветов (квадраты с `data-color="#hex"`).
- При выборе палитры — подставлять выбранный hex в скрытый input и в превью; при «Сбросить» — очищать значение (null) и отправлять в бэкенд пустое поле.
- Страница/модалка без тяжёлых библиотек — только Vanilla JS при необходимости (например, синхронизация input[type=color] и кнопок палитры).

---

## 4. Краткая схема потока данных

1. **Пользователь открывает «Мои репетиторы» / карточку репетитора / «Мои ученики» (группа):** отображается форма «Цвет» с текущим значением (если есть).
2. **Пользователь выбирает цвет и сохраняет:** POST на `update_tutor_color` или `update_group_color` → сохранение в ConnectionRequest или UserGroupColor.
3. **Пользователь открывает расписание или ДЗ:** view подгружает уроки/ДЗ и словари цветов (tutor_colors, group_colors) без N+1; в контексте для каждого элемента задаётся `display_color`.
4. **Шаблон рендерит карточки/строки:** при наличии `display_color` выставляет `--custom-entity-color` и класс индикатора; CSS рисует рамку и лёгкий фон.
5. **Сброс цвета:** в форме «Убрать цвет» отправляется пустое значение; в БД сохраняется null; при следующей загрузке расписания/ДЗ индикатор не показывается.

# Архитектура: расширения (оплаты, теги файлов, UI)

Ниже — модели и решения для пакета улучшений: система подтверждения оплат, тегирование файлов, фронтенд (сортировка, Yandex Maps).

## 1. Модели данных

### 1.1 PaymentReceipt (чеки об оплате)

Модель для загрузки чека учеником и верификации репетитором.

| Поле | Тип | Описание |
|------|-----|----------|
| student | FK → Users | Ученик, загрузивший чек |
| tutor | FK → Users | Репетитор, которому предназначена оплата |
| amount | DecimalField | Сумма (max_digits=10, decimal_places=2) |
| receipt_date | DateField | Дата платежа по чеку |
| file | FileField | Скан/фото чека (upload_to, валидатор размера/типа) |
| status | CharField | pending / approved / rejected |
| created_at | DateTimeField | Дата создания заявки |
| reviewed_at | DateTimeField, null | Когда репетитор подтвердил/отклонил |

- Ограничение: одна связь student–tutor; при необходимости unique_together не задавать (разрешено несколько чеков за разные даты).
- При status=approved создаётся Transaction (deposit) и обновляется StudentBalance в той же транзакции БД.

### 1.2 FileTag и связь с FilesLibrary

| Модель | Поля | Описание |
|--------|------|----------|
| FileTag | tutor (FK → Users), name (CharField) | Тег принадлежит репетитору; уникальность (tutor, name). |
| FilesLibrary | без изменений полей | Добавить M2M: tags = ManyToManyField(FileTag, blank=True, related_name='files'). |

Либо промежуточная таблица через M2M по умолчанию (FilesLibrary.tags → FileTag). Репетитор видит и управляет только своими тегами и файлами.

## 2. Бэкенд: транзакционность при подтверждении платежа

При действии репетитора «Подтвердить» выполнять в одной транзакции:

1. Проверка: чек в статусе pending, текущий пользователь — tutor этого чека.
2. `receipt.status = 'approved'`; `receipt.reviewed_at = timezone.now()`; `receipt.save()`.
3. `StudentBalance`: get_or_create(tutor, student), затем `balance += receipt.amount`; save().
4. `Transaction.objects.create(student=..., tutor=..., amount=receipt.amount, type='deposit', description='Пополнение по чеку от ...', date=...)`.

Всё в одном `with transaction.atomic():`. При любом исключении откат целиком.

При «Отклонить»: только обновление статуса и reviewed_at, без изменения баланса и без создания Transaction.

## 3. Фронтенд

### 3.1 Сортировка таблиц

- Универсальная функция на Vanilla JS: по клику на заголовок столбца определять индекс столбца и направление (asc/desc), сортировать строки таблицы (получать tbody.rows, сортировать по cell[index].textContent или data-атрибуту для чисел/дат), переставлять узлы в DOM.
- Для числовых и датовых столбцов — сравнение по числу/дате, а не по строке (при необходимости data-sort на ячейке).
- Визуальная индикация: класс на th (например .sort-asc / .sort-desc) и иконка/стрелка.
- Таблицы: расписание на index (core и mobile при наличии таблицы), файлы (если выводятся таблицей или сеткой с общим контейнером строк), журнал в student_card, finances.

### 3.2 Yandex Maps

- В шаблоне (модалка урока и при необходимости другие места) для локации урока:
  - Если `location` пустое — не показывать блок карт.
  - Если `location` начинается с `http://` или `https://` — считать ссылкой: не показывать кнопку карт, оставить копирование/ссылку.
  - Иначе считать адресом: вывести кнопку «Открыть в Яндекс.Картах» со ссылкой:
    - `https://yandex.ru/maps/?text={{ location|urlencode }}`
  - В Django шаблоне: `|urlencode` или кастомный фильтр/тег при необходимости корректного кодирования.

### 3.3 Hover и цветовая индикация

- Убрать из строк таблицы уроков inline `onmouseover`/`onmouseout`, задающие фиксированный background.
- В CSS: для `.entity-color-indicator:hover` задать фон через color-mix с той же `--custom-entity-color` и большей долей (например 18–20%), чтобы при наведении цвет сохранялся, но подсветка была заметна.
- Правило `tbody tr:hover` в style.css не должно перебивать строки с entity-color-indicator: использовать селектор вида `tbody tr:not(.entity-color-indicator):hover` или задать для `.entity-color-indicator:hover` более специфичное правило с тем же color-mix.

### 3.4 Цвет репетитора для связи с учеником

- В модели ConnectionRequest добавить поле `tutor_color_hex` (CharField, max_length=7, null=True, blank=True). Цвет, который задаёт репетитор для этой связи (для отображения в своём расписании).
- В представлениях расписания для репетитора: при отсутствии группы у урока брать цвет из ConnectionRequest по паре (tutor=user, student=lesson.student) — поле tutor_color_hex; при наличии группы — по UserGroupColor как сейчас.
- UI: в карточке ученика (student_card) для репетитора — блок «Цветовая метка (ваш цвет)» с палитрой и сохранением в tutor_color_hex (POST view или существующий endpoint с расширением).
