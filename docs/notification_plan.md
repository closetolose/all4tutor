# План расширения уведомлений

## 1. Текущее состояние

| Компонент | Описание |
|-----------|----------|
| **Модель** | `core.models.Notification` — user, message, link, is_read, type (warning/info), created_at |
| **Создание** | Только команда `check_overdue_homeworks`: уведомление ученику о просроченном ДЗ |
| **Показ** | Контекст-процессор `notifications_processor` → `unread_notifications_count`, `recent_notifications` (последние 10) |
| **UI** | Колокольчик в шапке (desktop + mobile), модалка со списком уведомлений |
| **API** | `api_notification_mark_read` — пометить уведомление прочитанным |

---

## 2. Цели расширения

- Добавить in-app уведомления для ключевых событий (ДЗ, чеки, заявки).
- Иметь единую точку создания уведомлений (хелпер) для удобства и опциональной отправки в Telegram.
- При желании — напоминания об уроках и кнопка «Прочитать всё».

---

## 3. Задачи

### 3.1. Хелпер создания уведомлений (обязательно)

**Задача:** Вынести создание уведомления в одну функцию.

- **Файл:** `core/utils.py` (или новый `core/services/notifications.py`).
- **Функция:** `notify_user(user, message, link=None, notification_type='info')`:
  - создаёт запись `Notification` для переданного `user` (берём `user` как `User`, не Profile);
  - при необходимости опционально вызывает `send_telegram_notification(user.profile, message)` (флаг или настройка).
- **Импорт:** в местах вызова использовать эту функцию вместо прямого `Notification.objects.create`.

**Критерий готовности:** Есть одна функция `notify_user`, все новые уведомления создаются через неё.

---

### 3.2. Уведомления при создании ДЗ

**Задача:** Когда репетитор задаёт ДЗ, ученик получает in-app уведомление.

- **Место:** Вьюха создания ДЗ (поиск по `Homework.objects.create` или `add_homework`).
- **Действие:** После успешного сохранения ДЗ вызвать `notify_user(hw.student.user, message, link=..., type='info')`.
- **Текст:** Например: «Вам задали домашнее задание по {предмет}. Дедлайн: {дата}» или короче: «Новое ДЗ по {предмет}».
- **Ссылка:** `reverse('homework_detail', args=[hw.id])`.

**Критерий готовности:** Создание ДЗ из интерфейса приводит к появлению уведомления у ученика в колокольчике.

---

### 3.3. Уведомления при сдаче ДЗ на проверку

**Задача:** Когда ученик меняет статус ДЗ на «На проверке», репетитор получает уведомление.

- **Место:** Вьюха/логика смены статуса ДЗ на `submitted` (поиск по `status='submitted'` или `Homework` update).
- **Действие:** После смены статуса вызвать `notify_user(hw.tutor.user, message, link=..., type='info')`.
- **Текст:** Например: «{Имя ученика} сдал(а) ДЗ по {предмет} на проверку».
- **Ссылка:** `reverse('homework_detail', args=[hw.id])`.

**Критерий готовности:** Сдача ДЗ на проверку создаёт уведомление у репетитора.

---

### 3.4. Уведомления по чекам (заявки на пополнение)

**Задача:** Репетитор получает уведомление при отправке чека; ученик — при подтверждении или отклонении.

| Событие | Кому | Место | Текст (пример) | Ссылка |
|---------|------|--------|----------------|--------|
| Ученик отправил чек | Репетитор | `submit_receipt` после `PaymentReceipt.objects.create` | «{Ученик} отправил чек на {сумма} ₽ на проверку» | `payment_receipts` |
| Репетитор подтвердил чек | Ученик | `approve_receipt` после успешного обновления баланса | «Чек подтверждён. Баланс пополнен на {сумма} ₽» | `payment_receipts` |
| Репетитор отклонил чек | Ученик | `reject_receipt` после сохранения статуса | «Чек отклонён» | `payment_receipts` |

- **Действие:** В каждом из трёх мест вызвать `notify_user(..., message, link=reverse('payment_receipts'))`.
- **user:** Для репетитора — `receipt.tutor.user`, для ученика — `receipt.student.user`.

**Критерий готовности:** Отправка чека, подтверждение и отклонение создают соответствующие уведомления.

---

### 3.5. Уведомления по заявкам на связь

**Задача:** Ученик получает уведомление при принятии/отклонении заявки репетитором.

- **Место:** Вьюхи принятия и отклонения заявки (поиск по `accept_request`, `reject_request` или обновлению `ConnectionRequest`).
- **При принятии:** `notify_user(connection.student.user, "Репетитор {имя} принял вашу заявку", link=reverse('tutor_card', args=[tutor.id]), type='info')`.
- **При отклонении:** `notify_user(connection.student.user, "Репетитор {имя} отклонил заявку", link=reverse('my_tutors'), type='warning')`.

**Критерий готовности:** Принятие и отклонение заявки создают уведомления у ученика.

---

### 3.6. Расширение типов уведомлений (опционально)

**Задача:** Добавить в модель типы для фильтрации и отображения (иконка/цвет).

- **Вариант A:** Оставить только `warning` и `info`, использовать их по смыслу.
- **Вариант B:** Добавить в `Notification.TYPE_CHOICES` значения, например: `success`, `danger`, или по сущностям: `homework`, `receipt`, `connection`, `lesson`.
- В шаблонах модалки колокольчика при выводе списка использовать `notification.type` для класса/иконки.

**Критерий готовности:** В админке и (при необходимости) в UI отображаются новые типы.

---

### 3.7. Напоминания об уроках (опционально)

**Задача:** За N часов до урока создавать уведомление ученику и репетитору.

- **Реализация:** Новая management-команда (например `send_lesson_reminders`):
  - выборка уроков с `start_time` в интервале [now + N часов − 1 час, now + N часов] (или за сутки);
  - для каждого урока: создать два уведомления — репетитору и ученику (для группового — можно одному уведомлению на ученика или на группу, по решению).
- **Текст:** «Через N часов занятие по {предмет}» (или «Занятие завтра в {время}»).
- **Ссылка:** `reverse('index')` или страница урока, если есть.
- **Запуск:** По расписанию (cron), например раз в час: `python manage.py send_lesson_reminders --hours 2`.

**Критерий готовности:** Команда создаёт уведомления; при добавлении в cron пользователи получают напоминания.

---

### 3.8. Кнопка «Прочитать всё» (опционально)

**Задача:** В модалке уведомлений добавить кнопку «Прочитать всё» и API для массовой пометки.

- **Backend:** Новая вьюха (например `api_notification_mark_all_read`): POST, для текущего пользователя выполнить `Notification.objects.filter(user=request.user).update(is_read=True)`, вернуть JSON `{ok: true}`.
- **URL:** Добавить в `urls.py` маршрут к этой вьюхе.
- **Frontend:** В шаблоне модалки колокольчика — кнопка «Прочитать всё», по клику — запрос к новому API, затем обновить счётчик (или перезагрузить страницу / обновить блок).

**Критерий готовности:** По нажатию «Прочитать всё» все уведомления пользователя помечаются прочитанными, бейдж на колокольчике сбрасывается.

## 5. Технические замечания

- **user в Notification:** Модель привязана к `User` (AUTH_USER_MODEL). В коде часто есть `user_profile` (Profile); для вызова передавать `user_profile.user`.
- **Telegram:** В хелпере можно опционально вызывать `send_telegram_notification(user.profile, message)` (проверять наличие `user.profile.telegram_id`), чтобы дублировать важные события в Telegram.
- **Дедупликация:** При необходимости избежать спама (например, много ДЗ подряд) можно ограничить частоту: не создавать новое уведомление одному пользователю по одному и тому же типу события, если уже есть непрочитанное за последние N минут (опционально, не в первой итерации).
- **Cron:** Для команд уведомлений (просроченные ДЗ, напоминания об уроках) настроить запуск по расписанию (cron или Celery beat).
